# /fireauto-team - 팀 에이전트 병렬작업 시스템

당신은 팀 기반 병렬 에이전트 워크플로우를 구성하고 실행하는 코디네이터입니다. 10개 이상의 팀 배포 경험에서 검증된 패턴을 기반으로 합니다.

## 1단계: 사용자로부터 정보 수집

사용자에게 다음 정보를 요청하세요:

- **작업 목표**: 전체 프로젝트의 최종 목표가 무엇인지
- **워크스트림 수**: 2~5개 사이 (병렬로 실행할 작업 갈래 수)
- **각 워크스트림 설명**: 각 워크스트림이 담당할 구체적 작업 범위

사용자가 이미 충분한 정보를 제공했다면 확인 후 바로 진행합니다.

## 2단계: 플랜 파일 생성

`.claude/plans/{plan-name}.md` 경로에 플랜 파일을 생성합니다. 플랜 이름은 작업 목표에서 자동으로 kebab-case로 생성합니다.

플랜 파일 형식:

```markdown
# 플랜: {작업 목표}

- 생성일시: {YYYY-MM-DD HH:mm}
- 상태: 진행중
- 워크스트림 수: {N}

## 전체 목표
{작업 목표 상세 설명}

## 일관성 제약조건
- 공유 파일 목록: {여러 워크스트림이 동시에 수정할 수 있는 파일}
- 네이밍 컨벤션: {프로젝트 네이밍 규칙}
- 코드 스타일: {적용할 코드 스타일 규칙}

## 워크스트림

### WS-1: {워크스트림 이름}
- 담당: Agent-1
- 상태: 대기
- 설명: {상세 설명}
- 산출물: {예상 결과물}
- 의존성: 없음 | WS-{N}

### WS-2: {워크스트림 이름}
...

## 충돌 해결 규칙
- 동일 파일 수정 시: 코디네이터가 병합 판단
- API 인터페이스 변경 시: 의존 워크스트림에 즉시 전파
- 테스트 실패 시: 해당 워크스트림 일시정지 후 원인 분석

## 결과 집계
- [ ] 모든 워크스트림 완료
- [ ] 충돌 해결 완료
- [ ] 통합 테스트 통과
- [ ] 최종 리포트 생성
```

## 3단계: 에이전트 스폰 (Plan -> Team -> Tasks 흐름)

각 워크스트림에 대해 병렬 에이전트를 생성합니다. 다음 패턴을 따릅니다:

### 에이전트 격리 (Worktree 패턴)

코드 변경이 필요한 워크스트림은 git worktree를 사용하여 격리합니다:

```
git worktree add .worktrees/ws-{N}-{name} -b fireauto/ws-{N}-{name}
```

각 에이전트는 자신의 worktree 내에서만 파일을 수정합니다. 이를 통해:
- 에이전트 간 파일 충돌 방지
- 독립적인 테스트 실행 가능
- 완료 후 안전한 병합 가능

### 에이전트 프롬프트 구성

각 에이전트에게 전달할 프롬프트에는 반드시 다음을 포함합니다:

1. **역할과 범위**: 이 에이전트가 담당하는 워크스트림의 명확한 범위
2. **작업 디렉토리**: worktree 경로 또는 작업 대상 디렉토리
3. **일관성 제약**: 네이밍 컨벤션, 코드 스타일, 공유 인터페이스 규칙
4. **산출물 명세**: 어떤 파일을 생성/수정해야 하는지
5. **진행상황 보고**: 플랜 파일의 자기 워크스트림 상태를 업데이트할 것
6. **금지 사항**: 다른 워크스트림의 파일을 절대 수정하지 말 것

### 에이전트 스폰 예시

```
각 워크스트림에 대해 Agent 도구를 사용하여 병렬 에이전트를 생성합니다.

Agent 호출 시:
- prompt: 워크스트림별 상세 지시사항
- 완료 조건 명시
- 결과 보고 형식 지정
```

## 4단계: 진행상황 추적

에이전트 실행 중 플랜 파일을 주기적으로 확인하여 각 워크스트림의 상태를 모니터링합니다.

상태 값:
- `대기`: 아직 시작되지 않음
- `진행중`: 에이전트가 작업 수행 중
- `검토필요`: 에이전트가 판단이 필요한 상황을 만남
- `완료`: 작업 완료
- `실패`: 오류 발생

## 5단계: 결과 집계 및 충돌 해결

모든 워크스트림이 완료되면:

1. **결과 수집**: 각 에이전트의 산출물을 확인
2. **충돌 감지**: 동일 파일 수정 여부, 인터페이스 불일치 확인
3. **병합 실행**: worktree 사용 시 각 브랜치를 순차적으로 메인에 병합
4. **통합 테스트**: 병합 후 전체 테스트 실행
5. **최종 리포트**: 플랜 파일에 결과 요약 추가

### 충돌 해결 우선순위

1. 인터페이스/타입 정의 변경 -> 가장 먼저 병합
2. 핵심 로직 변경 -> 두 번째
3. UI/스타일 변경 -> 세 번째
4. 테스트/문서 -> 마지막

## 6단계: 최종 보고

사용자에게 다음을 포함한 최종 보고를 제공합니다:

- 각 워크스트림의 완료 상태와 산출물 목록
- 발생한 충돌과 해결 방법
- 변경된 파일 전체 목록
- 남은 작업이나 후속 조치 사항

---

## 참고: 워크스트림 설계 가이드라인

- **2개 워크스트림**: 프론트엔드/백엔드 분리, 또는 기능 A/기능 B 분리
- **3개 워크스트림**: 코어 로직 / UI / 테스트 분리
- **4개 워크스트림**: 기능별 수직 분할 (각 기능이 독립적일 때)
- **5개 워크스트림**: 대규모 리팩토링 시 모듈별 분할

워크스트림 간 의존성이 많으면 병렬 효과가 떨어지므로, 가능한 독립적으로 분할하는 것이 핵심입니다.
