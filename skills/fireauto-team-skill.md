# 팀 에이전트 패턴 스킬

## 개요

팀 에이전트는 복잡한 작업을 여러 워크스트림으로 분할하여 병렬로 실행하는 패턴입니다. 각 워크스트림은 독립된 에이전트가 담당하며, 코디네이터가 전체를 조율합니다.

## 병렬 vs 순차 판단 기준

### 병렬 작업이 적합한 경우

- **독립적인 기능 개발**: 서로 다른 페이지, 컴포넌트, API 엔드포인트를 동시에 개발
- **프론트엔드/백엔드 분리**: 인터페이스 계약이 명확할 때 양쪽을 동시 개발
- **대규모 리팩토링**: 모듈별로 분리하여 각각 리팩토링
- **테스트 작성**: 기존 코드에 대한 테스트를 여러 영역으로 나누어 동시 작성
- **다국어 콘텐츠**: 여러 언어의 번역을 동시 진행
- **마이그레이션**: 여러 데이터 소스나 서비스를 동시에 마이그레이션

### 순차 작업이 적합한 경우

- **강한 의존성**: 앞선 작업의 결과물이 다음 작업의 입력이 되는 경우
- **동일 파일 집중 수정**: 같은 파일을 여러 관점에서 수정해야 하는 경우
- **설계 -> 구현 -> 테스트**: 단계적으로 진행해야 하는 경우
- **데이터베이스 스키마 변경**: 스키마 변경 후 코드 수정이 필요한 경우
- **작은 규모의 작업**: 분할 오버헤드가 작업 자체보다 큰 경우

## 워크스트림 설계 원칙

### 1. 경계 명확화
각 워크스트림의 책임 범위를 명확히 정의합니다. "이 워크스트림은 이 디렉토리/파일만 수정한다"와 같이 물리적 경계를 설정하는 것이 가장 효과적입니다.

### 2. 인터페이스 선행 정의
워크스트림 간 공유하는 인터페이스(타입, API 스키마, props)를 작업 시작 전에 먼저 정의합니다. 이 인터페이스는 코디네이터가 관리합니다.

### 3. 적절한 분할 크기
- 2개: 가장 안정적. 프론트/백 또는 코어/UI 분리에 적합
- 3개: 코어/UI/테스트, 또는 3개 독립 기능에 적합
- 4-5개: 대규모 작업에만 사용. 관리 복잡도 증가를 감안

### 4. 의존성 최소화
워크스트림 간 의존성이 많으면 병렬의 이점이 사라집니다. 의존성이 있다면 선행 워크스트림이 인터페이스를 먼저 확정한 후 후속 워크스트림을 시작합니다.

## 에이전트 격리 패턴

### Git Worktree 패턴 (권장)
```bash
# 워크스트림별 독립 작업 공간 생성
git worktree add .worktrees/ws-1-frontend -b fireauto/ws-1-frontend
git worktree add .worktrees/ws-2-backend -b fireauto/ws-2-backend

# 작업 완료 후 병합
git checkout main
git merge fireauto/ws-1-frontend
git merge fireauto/ws-2-backend

# 정리
git worktree remove .worktrees/ws-1-frontend
git worktree remove .worktrees/ws-2-backend
```

### 디렉토리 분할 패턴 (간단한 경우)
코드 변경 없이 분석, 문서 작성 등의 작업이라면 worktree 없이 디렉토리만 분할하여 작업할 수 있습니다.

## 일관성 제약 관리

병렬 작업 시 다음 항목의 일관성을 유지해야 합니다:

1. **네이밍 컨벤션**: 변수명, 함수명, 파일명 규칙을 사전에 통일
2. **코드 스타일**: 포매터, 린터 설정을 공유
3. **타입 정의**: 공유 타입을 별도 파일로 관리하고, 한 워크스트림만 수정 권한 부여
4. **API 계약**: 엔드포인트 URL, 요청/응답 형식을 사전 합의
5. **상태 관리**: 전역 상태 구조를 사전에 설계

## 진행 추적 체계

### 플랜 파일 (.claude/plans/)
전체 작업 계획과 각 워크스트림의 상태를 기록합니다. 코디네이터와 각 에이전트 모두 이 파일을 참조합니다.

### 상태 업데이트 규칙
각 에이전트는 자신의 워크스트림 상태를 플랜 파일에서 직접 업데이트합니다:
- 작업 시작 시: `대기` -> `진행중`
- 판단 필요 시: `진행중` -> `검토필요` + 사유 기록
- 작업 완료 시: `진행중` -> `완료` + 산출물 목록
- 오류 발생 시: `진행중` -> `실패` + 오류 내용

## 결과 집계 및 충돌 해결

### 병합 순서
1. 인터페이스/타입 변경이 포함된 워크스트림을 먼저 병합
2. 핵심 로직 변경을 다음으로 병합
3. UI/프레젠테이션 변경을 이후 병합
4. 테스트/문서를 마지막으로 병합

### 충돌 해결 전략
- **자동 해결 가능**: import 추가, export 목록 확장 등 -> 자동 병합
- **수동 해결 필요**: 동일 함수 수정, 로직 충돌 -> 코디네이터가 판단
- **구조적 충돌**: 아키텍처 수준 불일치 -> 사용자에게 보고 후 판단 요청

## 실전 활용 사례

### 사례 1: 랜딩페이지 리뉴얼 (3 워크스트림)
- WS-1: 히어로 섹션 + 네비게이션 리디자인
- WS-2: 가격/기능 섹션 리디자인
- WS-3: 반응형 + 접근성 개선

### 사례 2: API + 프론트 동시 개발 (2 워크스트림)
- WS-1: 백엔드 API 엔드포인트 구현
- WS-2: 프론트엔드 UI + API 연동 (목업 데이터로 시작)

### 사례 3: 대규모 리팩토링 (4 워크스트림)
- WS-1: 타입 시스템 정리 및 공유 인터페이스 정의
- WS-2: 컴포넌트 구조 리팩토링
- WS-3: 상태 관리 마이그레이션
- WS-4: 테스트 코드 업데이트

## 사용 방법

팀 에이전트 워크플로우를 시작하려면 `/fireauto-team` 명령어를 사용하세요.
진행 상태를 확인하려면 `/fireauto-team-status` 명령어를 사용하세요.
